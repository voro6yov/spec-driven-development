# Messaging Updates Report Template

> **Consumers:**
> - `messaging-updates-writer` agent — renders the report; uses these rules to compute the per-consumer change blocks and the `## Affected Artifacts` footer.
> - `/messaging-spec:update-code` skill (future) — parses the report to dispatch per-consumer code edits.

> **Scope of this skill:** output format only. Workflow (discovering consumers, diffing each consumer spec against `git HEAD`, parsing the domain `updates.md`, computing the affected / aborted sets, rendering) lives in the `messaging-updates-writer` agent body.

---

## What makes this report different

There is no single spec file. `<dir>/<stem>.messaging/` holds *N* consumer specs (`<consumer-name>.md`), and a domain change touches a subset of them. So this report is **consumer-keyed**: `## Consumer Changes` lists every discovered consumer with a status (`updated` | `aborted` | `unaffected`), and `## Affected Artifacts` lists the generated files of the `updated` consumers only.

The only domain-sensitive part of a consumer spec is Table 3's `Event Field` column for `internal`-event sub-blocks — so the per-consumer change body is small (a list of regenerated Table 3 sub-blocks). External events never appear here (their classes live on the commands diagram, not the domain diagram), and Table 1 / Table 2 are never regenerated by a domain-driven change (Table 2 may go *stale* — that is the `aborted` case).

---

## Schema

Substitute every `<placeholder>` with the actual value when rendering. The `### \`<consumer_name>\` — <status>` block recurs once per discovered consumer; the three forms below show each status's body.

````markdown
<!-- domain-updates-hash:<hash> -->

# Messaging Updates Report

## Summary

- Messaging folder: `<dir>/<stem>.messaging/`
- Commands diagram: `<dir>/<stem>.commands.md`
- Domain updates source: `<dir>/<stem>.domain/updates.md` (hash: <sha256>)
- Consumers discovered: <N>
- Consumers updated: <N>
- Consumers aborted (reconcile commands diagram): <N>
- Consumers unaffected: <N>
- Warnings:
  - <warning text>

## Consumer Changes

### `<consumer_name>` — updated

- Spec: `<dir>/<stem>.messaging/<consumer_name>.md`
- Pre-update hash: <sha256>
- Post-update hash: <sha256>
- Table 2 refreshed: <row delta phrase>
- Table 3 sub-blocks regenerated:
  - `**Event:** <EventName>` (internal · source `<SourceDestination>`)
    - Source delta: <short_phrase>
    - Event Field mappings changed:
      - row `<command_param>` ↦ `<event_field>` added
      - row `<command_param>` ↦ `<event_field>` removed
      - row `<command_param>`: `<event_field_old>` → `<event_field_new>` changed
    - Low-confidence flags: <verbatim italic flag text> | _none_

### `<consumer_name>` — aborted (reconcile commands diagram)

- Spec: `<dir>/<stem>.messaging/<consumer_name>.md` (unchanged this run)
- Dangling internal event(s): `<EventName>` (removed or renamed in `<dir>/<stem>.domain/updates.md`)
- Required reconcile:
  - Edit the `%% Messaging - <consumer_name>` block in `<dir>/<stem>.commands.md`: drop or rename the `<CommandClass> --() <EventName> : handles (<SourceDestination>, <on_method>)` line, and the `<on_method>` handler on `<CommandClass>`.
  - Then re-run `event-tables-writer <dir>/<stem>.commands.md <consumer_name>` and `event-fields-writer <dir>/<stem>.commands.md <consumer_name>` (or `/messaging-spec:generate-code <dir>/<stem>.md <consumer_name>`).

### `<consumer_name>` — unaffected

_No internal subscription intersects the changed-domain-event set._

## Affected Artifacts

| Path | Action | Driving consumer |
|---|---|---|
| `messaging/<consumer_snake>/handlers.py` | modify | `<consumer_name>` |
| `tests/integration/messaging/<consumer_snake>/test_<consumer_snake>_handlers.py` | modify | `<consumer_name>` |
````

## Rendering rules

### Top-of-file sentinel

The first line of the file is an HTML comment recording the SHA256 of `<dir>/<stem>.domain/updates.md`:

```
<!-- domain-updates-hash:<sha256> -->
```

When `<stem>.domain/updates.md` does not exist on disk, render `<sha256>` as `(none)`. The sentinel line itself is always emitted on line 1, followed by one blank line, then the `# Messaging Updates Report` heading.

The sentinel is the consumer's primary skip-on-replay signal: a downstream `/messaging-spec:update-code` run that already applied a report carrying the same `domain-updates-hash` may early-exit.

### Top-level sections

All three sections are **always emitted** with their headings, in this canonical order:

1. `## Summary`
2. `## Consumer Changes`
3. `## Affected Artifacts`

`## Summary` always carries its bullet list (never `_no changes_`). `## Consumer Changes` always lists **every discovered consumer** (one H3 block each); when zero consumers exist its body is the single literal line `_no consumers_`. `## Affected Artifacts` always emits the table header; its data rows are empty when no consumer is `updated`.

### Consumer ordering

Inside `## Consumer Changes`, H3 blocks are ordered **alphabetically by consumer name**, regardless of status — `aborted`, `updated`, and `unaffected` blocks interleave by name (no status grouping). The consumer name is the spec file's basename with `.md` stripped, rendered verbatim inside backticks in the H3 heading. The same name (verbatim, backticked) is the `Driving consumer` cell in the footer.

### H3 heading form and the status vocabulary

Each consumer's block is headed `### \`<consumer_name>\` — <status>` where `<status>` is exactly one of this closed vocabulary:

| Status (verbatim heading suffix) | When |
|---|---|
| `updated` | The consumer's Table 3 has at least one `internal`-event sub-block whose row set or low-confidence italic-flag text differs between the HEAD and working-tree versions of the consumer spec **for a domain event the domain `updates.md` reports as having an attribute add/remove/rename**, **and** the consumer is not `aborted` and not newly-tracked-this-run. (A Table 3 sub-block change whose event has no matching domain attribute delta is byte-stable in practice; if it isn't, the cause is a commands-diagram edit — a separate axis, out of scope here.) |
| `aborted (reconcile commands diagram)` | The consumer subscribes (as `internal`, per its Table 2) to a domain event that the domain `updates.md` reports as removed or renamed. The spec file is unchanged this run. Wins over `updated` / `unaffected`. |
| `unaffected` | None of the above. Includes: (a) the consumer subscribes to no domain event that changed this run; (b) it *does* subscribe (as `internal`) to a changed-attribute domain event but the bound handler does not consume the changed attribute, so Table 3 came out byte-stable (→ use the precise body form below); (c) the consumer spec is newly tracked this run, with no HEAD blob (→ a "first-run consumer" warning is emitted; its Table 3 was just *generated*, not *updated*). |

A consumer is never simultaneously `updated` and `aborted` — if any of its `internal` subscriptions is dangling, the whole consumer is `aborted` for this run; its other subscriptions wait for the next run after the operator reconciles the commands diagram. A newly-tracked consumer never reports as `updated` — a fresh Table 3 is a generation, not an update.

### `updated` block body

Bullets, in this fixed order (omit a bullet only where the rule below says so):

- **`Spec:`** — the consumer spec path.
- **`Pre-update hash:`** / **`Post-update hash:`** — per the Hash format rule below; these always differ for an `updated` block.
- **`Table 2 refreshed:`** — emitted **only when** the consumer spec's Table 2 content differs between HEAD and the working tree (e.g. the operator already reconciled the commands diagram before the run, so the optional `event-tables-writer` re-run picked it up). The value is a short phrase describing the row delta — `added: <rows>; removed: <rows>; changed: <rows>`, with empty clauses dropped. On a pure domain-axis change Table 2 is byte-stable, so this bullet is **absent**.
- **`Table 3 sub-blocks regenerated:`** — a bullet list, one entry per regenerated sub-block. A sub-block is listed iff its row set or its low-confidence italic-flag text differs between the HEAD and working-tree versions of the consumer spec **and** its event is one the domain `updates.md` reports as having an attribute add/remove/rename. Only `internal` events ever appear here (the domain axis never touches `external` rows; and an `internal` sub-block change for an event with no domain attribute delta is byte-stable in practice — out of scope). Order the entries like Table 3 (`external` block alphabetical, then `internal` block alphabetical — in practice alphabetical by Event Name). Each entry:
  - **Heading line:** `` `**Event:** <EventName>` (internal · source `<SourceDestination>`) `` — `<SourceDestination>` is read from the matching `internal` Table 2 row of the post-update spec.
  - **`Source delta:`** — a short phrase derived from `<dir>/<stem>.domain/updates.md`, of the form `domain-events: <EventClass> attribute <attr> added` / `domain-events: <EventClass> attribute <attr> removed` / `domain-events: <EventClass> attribute <attr> renamed to <new>`. When several attribute deltas affect the same event, join them with `; `. When the domain `updates.md` is missing or carries no matching entry, the literal string `(unknown source)`.
  - **`Event Field mappings changed:`** — a sub-bullet list of the row-level changes between the HEAD and working-tree Table 3 sub-blocks, keyed by Command Parameter:
    - `row \`<param>\` ↦ \`<event_field>\` added` — a Command Parameter row present in the working tree but not HEAD.
    - `row \`<param>\` ↦ \`<event_field>\` removed` — a row present in HEAD but not the working tree (`<event_field>` is its HEAD value).
    - `row \`<param>\`: \`<old>\` → \`<new>\` changed` — a Command Parameter present in both whose Event Field value differs.
    When the row set is byte-identical (only the italic flag was re-derived), render the single sub-bullet `_(none — only the low-confidence flag was re-derived)_`.
  - **`Low-confidence flags:`** — the working-tree italic-prose flag(s) `event-fields-writer` emitted in/around this sub-block, rendered **verbatim** (markdown emphasis markers stripped, leading/trailing whitespace trimmed), joined with `; `. When the sub-block carries no italic flag, the literal `_none_`.

### `aborted` block body

Bullets, in this fixed order:

- **`Spec:`** — the consumer spec path with the literal suffix ` (unchanged this run)`. (`Pre-update hash` / `Post-update hash` are **omitted** from an `aborted` block — the spec is unchanged this run.)
- **`Dangling internal event(s):`** — comma-separated backticked event names (each ∈ the domain report's removed-or-renamed-event set ∩ this consumer's `internal` Table 2 rows), followed by ` (removed or renamed in \`<dir>/<stem>.domain/updates.md\`)`.
- **`Required reconcile:`** — a two-sub-bullet instruction:
  - `Edit the \`%% Messaging - <consumer_name>\` block in \`<dir>/<stem>.commands.md\`: drop or rename the \`<CommandClass> --() <EventName> : handles (<SourceDestination>, <on_method>)\` line, and the \`<on_method>\` handler on \`<CommandClass>\`.` — the `<CommandClass>` / `<SourceDestination>` / `<on_method>` tokens come from the consumer spec's stale Table 2 row for the dangling event. When more than one event is dangling, repeat the `<CommandClass> --() <EventName> : handles (...)` clause for each, separated by `; `.
  - `Then re-run \`event-tables-writer <dir>/<stem>.commands.md <consumer_name>\` and \`event-fields-writer <dir>/<stem>.commands.md <consumer_name>\` (or \`/messaging-spec:generate-code <dir>/<stem>.md <consumer_name>\`).`

### `unaffected` block body

A single italic line, and nothing else — one of two forms:

- **Standard** — when the consumer subscribes to no domain event that changed this run (the *changed-domain-event set* = removed-or-renamed events ∪ events with an attribute add/remove/rename):

  ```
  _No internal subscription intersects the changed-domain-event set._
  ```

- **Precise (byte-stable affected)** — when the consumer *does* subscribe (as `internal`) to a domain event that had an attribute add/remove/rename this run, but its Table 3 came out byte-stable (the bound handler does not consume the changed attribute):

  ```
  _Subscribes to changed domain event(s) `<EventName>`, but Table 3 is byte-stable (the bound handler does not consume the changed attribute)._
  ```

  List every such event in `<EventName>` — comma-separated, each backticked. This precise form is the **only** place the byte-stable-affected case is surfaced; there is no corresponding `Warnings:` entry.

A newly-tracked consumer (no HEAD blob) is `unaffected` too and uses whichever of the two forms applies (precise when it subscribes to a changed event, standard otherwise); a separate "first-run consumer" warning marks it as freshly generated rather than updated. Pick exactly one form; never mix them, never add other bullets to an `unaffected` block.

### Section: Summary

- The seven lines **Messaging folder**, **Commands diagram**, **Domain updates source**, **Consumers discovered**, **Consumers updated**, **Consumers aborted (reconcile commands diagram)**, **Consumers unaffected** are always emitted, in that order.
- The identity `updated + aborted + unaffected = discovered` always holds.
- **Domain updates source** value is `_none_` when no domain `updates.md` exists; otherwise the path plus a parenthesised hash (`<dir>/<stem>.domain/updates.md (hash: <sha256>)`).
- The **Warnings** line is omitted entirely when there are no warnings. When present it introduces a sub-bullet list. Warning categories (each rendered only when applicable; order as listed):
  - Dead subscription: `` `<consumer_name>` subscribes to internal event `<EventName>`, which is no longer emitted by `<AggregateRoot>` (dead subscription — byte-stable spec) `` — emitted when the domain `updates.md` records a `: emits <EventName>` edge removal whose event class survives and some consumer subscribes to it as `internal`. Best-effort: skipped when the domain `updates.md` is missing or carries no parseable relationship-removal entry.
  - Subscription candidate: `` domain event `<EventName>` was added; it is a subscription candidate (declare a `%% Messaging` marker in the commands diagram to consume it) `` — one per `<<Event>>` class added in the domain `updates.md`.
  - First-run consumer: `` `<consumer_name>` is newly tracked (no HEAD blob); its consumer spec was just generated, not updated by this run `` — one per consumer spec with no HEAD blob.
  - Domain updates source missing: `domain updates source not found; all source-delta values fell back to '(unknown source)' and no aborted consumers could be computed` — emitted when `<dir>/<stem>.domain/updates.md` is absent.
- Generated-at timestamps are **not** included — a wall-clock value would break the byte-stability contract.

## `## Affected Artifacts` computation

A flat dispatch table. The code updater walks it top-to-bottom. Only `updated` consumers contribute rows; `aborted` and `unaffected` consumers contribute nothing.

For each `updated` consumer `C` (consumer name `<C>`, snake form `<C_snake>` = `<C>` with every `-` replaced by `_`), emit, in this order:

1. `messaging/<C_snake>/handlers.py | modify | <C>`
2. `tests/integration/messaging/<C_snake>/test_<C_snake>_handlers.py | modify | <C>`

On the domain axis only `modify` ever appears — an internal-event attribute change re-renders an existing handler's call kwargs and that handler's test; it never adds or removes a handler (handler add/remove is a Table 2 / commands-diagram concern), and `dispatcher.py`, `events.py`, `constants.py`, and the `containers.py` / `entrypoint.py` / `__main__.py` wiring are byte-stable on this axis. The per-handler granularity (which handler inside `handlers.py`) lives in the consumer's `## Consumer Changes` body (the regenerated sub-block list), not in the footer.

### Row ordering

Consumers in the table follow the same alphabetical order as `## Consumer Changes`. Within a consumer's two rows, `handlers.py` precedes the test module.

### Action vocabulary

The `Action` column is a closed set: `add`, `modify`, `remove`. On the domain axis only `modify` is emitted. The `Driving consumer` column is the consumer name in backticks — the same anchor used as the H3 heading in `## Consumer Changes`.

The table header (`| Path | Action | Driving consumer |` plus the divider row) is always emitted. When no consumer is `updated`, the table has no data rows.

---

## Hash format

All hashes in this report are SHA256 of UTF-8 file content, rendered in **lowercase hex**, full **64-character** form.

When a hash cannot be computed (file missing or unreadable), render the value literally as `(none)`. Never substitute zeros.

---

## Determinism contract

- Byte-stable inputs (working-tree consumer specs, HEAD consumer specs, sibling `<dir>/<stem>.domain/updates.md`, sibling `<dir>/<stem>.commands.md`) → byte-stable report.
- Re-running the writer with no new domain changes produces a report whose every consumer is `unaffected`, with an empty Affected Artifacts row list and the same sentinel hash.
- Section ordering, consumer ordering (alphabetical, no status grouping), sub-block ordering (Table 3 order — `external` alphabetical then `internal` alphabetical, so in practice alphabetical by Event Name), and within-block bullet order are absolute. No source-defined deviation.
- The Summary deliberately excludes a `Generated at` line.
