# Messaging Updates Report Template

> **Consumers:**
> - `messaging-updates-writer` agent — renders the report; uses these rules to compute the per-consumer change blocks and the `## Affected Artifacts` footer.
> - `/messaging-spec:update-code` skill — parses the report to dispatch per-consumer code edits.

> **Scope of this skill:** output format only. Workflow (discovering consumers, diffing each consumer spec against `git HEAD`, parsing the domain `updates.md` and the commands-diagram `commands-updates.md`, computing the affected / aborted / needs-init / orphaned sets, rendering) lives in the `messaging-updates-writer` agent body.

---

## What makes this report different

There is no single spec file. `<dir>/<stem>.messaging/` holds *N* consumer specs (`<consumer-name>.md`), and a change touches a subset of them. So this report is **consumer-keyed**: `## Consumer Changes` lists every discovered consumer with a status (`updated` | `aborted` | `unaffected` | `needs-init` | `orphaned`), and `## Affected Artifacts` lists the generated files of the `updated` consumers only.

Three axes drive the writer: the **domain axis** (sibling `<dir>/<stem>.domain/updates.md`, drives `aborted` / `updated` / `unaffected`), the **commands-diagram axis** (sibling `<dir>/<stem>.application/commands-updates.md`), and the **ops-diagram axis** (sibling `<dir>/<stem>.application/ops-updates.md`). The two marker axes (commands and ops) drive `needs-init` / `orphaned` and contribute to `updated`-block `Source delta` attribution — they are peers, because a consumer's `%% Messaging - <C>` handler-binding block may live in the commands diagram or an ops diagram. All inputs are optional and individually non-fatal; absence of any silently skips that axis and surfaces a warning (a present-but-`_None_` ops report — zero ops diagrams — is a silent no-op, not a warning).

The only domain-sensitive part of a consumer spec is Table 3's `Event Field` column for `internal`-event sub-blocks — so an `updated` consumer's per-consumer change body is small (a list of regenerated Table 3 sub-blocks). External events live on the commands diagram; their attribute changes drive `external` sub-block regeneration only when the commands-diagram axis is present. Table 1 / Table 2 are never regenerated by a domain-driven change (Table 2 may go *stale* — that is the `aborted` case).

## Rendering rules

### Top-of-file sentinel block

The first three lines of the file are an HTML-comment block recording the SHA256 of each input updates source, one per axis:

```
<!-- domain-updates-hash:<sha256> -->
<!-- commands-updates-hash:<sha256> -->
<!-- ops-updates-hash:<sha256> -->
```

Order is fixed: domain, then commands-diagram, then ops-diagram. Each line is **always emitted**; when its source file is missing on disk, render `<sha256>` as the literal `(none)`. Never substitute zeros.

Render the three sentinel lines as the first three lines of the file, followed by one blank line, then the `# Messaging Updates Report` heading. (No additional blank line between the sentinel lines themselves.)

The sentinels are the consumer's primary skip-on-replay signal: a downstream `/messaging-spec:update-code` run that already applied a report carrying the same `(domain-updates-hash, commands-updates-hash, ops-updates-hash)` triple may early-exit. There is no queries-axis sentinel — messaging is command-side only.

### Top-level sections

The four data sections are emitted in this canonical order:

1. `## Summary` — always emitted with its bullet list.
2. `## Consumer Changes` — always emitted; lists every discovered consumer (one H3 block each). When zero consumers are discovered, its body is the single literal line `_no consumers_`.
3. `## Operator Actions` — emitted **only when** at least one consumer is `needs-init` or `orphaned`. When both lists are empty, omit the heading entirely (no `_no actions_` placeholder).
4. `## Affected Artifacts` — always emitted; emits its table header even when no consumer is `updated` (data rows empty).

### Consumer ordering

Inside `## Consumer Changes`, H3 blocks are ordered **alphabetically by consumer name**, regardless of status — `updated`, `aborted`, `unaffected`, `needs-init`, and `orphaned` blocks interleave by name (no status grouping). The consumer name is the spec file's basename with `.md` stripped (or, for a `needs-init` consumer, the verbatim consumer name from the commands-updates report's `## Messaging Markers` heading), rendered verbatim inside backticks in the H3 heading. The same name (verbatim, backticked) is the `Driving consumer` cell in the footer (when present) and the lead-token in each `## Operator Actions` bullet (when present).

### H3 heading form and the status vocabulary

Each consumer's block is headed `### \`<consumer_name>\` — <status>` where `<status>` is exactly one of this closed vocabulary:

| Status (verbatim heading suffix) | When |
|---|---|
| `updated` | The consumer's Table 3 has at least one `internal`-event sub-block whose row set or low-confidence italic-flag text differs between the HEAD and working-tree versions of the consumer spec **and** at least one probe (domain attr delta, commands marker change, external-event attr delta) explains the change, **and** the consumer is not `aborted` / `orphaned` / `needs-init` and not newly-tracked-this-run. |
| `aborted (reconcile commands diagram)` | The consumer subscribes (as `internal`, per its Table 2) to a domain event that the domain `updates.md` reports as removed or renamed, **and** the commands diagram still declares the subscription. The spec file is unchanged this run. (When the commands diagram already reconciled the dangling row, the consumer drops out of `aborted` and may surface as `orphaned` or `updated` instead.) |
| `unaffected` | None of the above. Includes: (a) the consumer subscribes to no domain event that changed this run; (b) it *does* subscribe (as `internal`) to a changed-attribute domain event but the bound handler does not consume the changed attribute, so Table 3 came out byte-stable (→ use the precise body form below); (c) the consumer spec is newly tracked this run, with no HEAD blob (→ a "first-run consumer" warning is emitted; its Table 3 was just *generated*, not *updated*). |
| `needs-init` | The commands diagram declares a `%% Messaging - <C>` block for which no consumer spec exists on disk. The commands-updates report lists `<C>` as `(consumer added)` under `## Messaging Markers`. The spec must be initialized via `@messaging-spec:code-generator <domain_diagram> <C>` before this consumer can participate in updates. |
| `orphaned` | A consumer spec exists on disk but the commands diagram no longer declares its `%% Messaging - <C>` block. The commands-updates report lists `<C>` as `(consumer removed)` under `## Messaging Markers`. The spec is unchanged this run. Operator action: preserve or delete the spec file, then re-run `@messaging-spec:code-generator <domain_diagram>` to reconcile the code side. |

Status precedence is `orphaned → aborted → needs-init → updated → unaffected` (first match wins). `orphaned` and `needs-init` are **advisory** — they describe a state mismatch between the commands diagram and the on-disk spec set, not a transition of a single file; they contribute zero rows to `## Affected Artifacts` and surface in `## Operator Actions` instead. A consumer is never simultaneously `updated` and `aborted` — if any of its `internal` subscriptions is dangling and the commands diagram still declares it, the whole consumer is `aborted` for this run; its other subscriptions wait for the next run after the operator reconciles. A newly-tracked consumer never reports as `updated` — a fresh Table 3 is a generation, not an update.

### `updated` block body

Bullets, in this fixed order (omit a bullet only where the rule below says so):

- **`Spec:`** — the consumer spec path.
- **`Pre-update hash:`** / **`Post-update hash:`** — per the Hash format rule below; these always differ for an `updated` block.
- **`Table 2 refreshed:`** — emitted **only when** the consumer spec's Table 2 content differs between HEAD and the working tree (e.g. the operator already reconciled the commands diagram before the run, so the optional `event-tables-writer` re-run picked it up). The value is a short phrase describing the row delta — `added: <rows>; removed: <rows>; changed: <rows>`, with empty clauses dropped. On a pure domain-axis change with no commands-diagram edits Table 2 is byte-stable, so this bullet is **absent**.
- **`Table 3 sub-blocks regenerated:`** — a bullet list, one entry per regenerated sub-block. A sub-block is listed iff its row set or its low-confidence italic-flag text differs between the HEAD and working-tree versions of the consumer spec **and** at least one source-delta probe (domain / commands-diagram) explains the change. Order the entries like Table 3 (`external` block alphabetical, then `internal` block alphabetical — in practice alphabetical by Event Name). Each entry:
  - **Heading line:** `` `**Event:** <EventName>` (internal · source `<SourceDestination>`) `` — `<SourceDestination>` is read from the matching Table 2 row of the post-update spec.
  - **`Source delta:`** — an axis-tagged phrase per the *Source delta phrase grammar* rule below; or the literal `(unknown source)` when no probe matches.
  - **`Event Field mappings changed:`** — a sub-bullet list of the row-level changes between the HEAD and working-tree Table 3 sub-blocks, keyed by Command Parameter:
    - `row \`<param>\` ↦ \`<event_field>\` added` — a Command Parameter row present in the working tree but not HEAD.
    - `row \`<param>\` ↦ \`<event_field>\` removed` — a row present in HEAD but not the working tree (`<event_field>` is its HEAD value).
    - `row \`<param>\`: \`<old>\` → \`<new>\` changed` — a Command Parameter present in both whose Event Field value differs.
    When the row set is byte-identical (only the italic flag was re-derived), render the single sub-bullet `_(none — only the low-confidence flag was re-derived)_`.
  - **`Low-confidence flags:`** — the working-tree italic-prose flag(s) `event-fields-writer` emitted in/around this sub-block, rendered **verbatim** (markdown emphasis markers stripped, leading/trailing whitespace trimmed), joined with `; `. When the sub-block carries no italic flag, the literal `_none_`.

### `aborted` block body

Bullets, in this fixed order:

- **`Spec:`** — the consumer spec path with the literal suffix ` (unchanged this run)`. (`Pre-update hash` / `Post-update hash` are **omitted** from an `aborted` block — the spec is unchanged this run.)
- **`Dangling internal event(s):`** — comma-separated backticked event names (each ∈ the domain report's removed-or-renamed-event set ∩ this consumer's `internal` Table 2 rows), followed by ` (removed or renamed in \`<dir>/<stem>.domain/updates.md\`)`.
- **`Required reconcile:`** — a two-sub-bullet instruction:
  - `Edit the \`%% Messaging - <consumer_name>\` block in \`<dir>/<stem>.commands.md\`: drop or rename the \`<CommandClass> --() <EventName> : handles (<SourceDestination>, <on_method>)\` line, and the \`<on_method>\` handler on \`<CommandClass>\`.` — the `<CommandClass>` / `<SourceDestination>` / `<on_method>` tokens come from the consumer spec's stale Table 2 row for the dangling event. When more than one event is dangling, repeat the `<CommandClass> --() <EventName> : handles (...)` clause for each, separated by `; `.
  - `Then re-run \`event-tables-writer <dir>/<stem>.commands.md <consumer_name>\` and \`event-fields-writer <dir>/<stem>.commands.md <consumer_name>\` (or \`@messaging-spec:code-generator <dir>/<stem>.md <consumer_name>\`).`

### `unaffected` block body

A single italic line, and nothing else — one of two forms:

- **Standard** — when the consumer subscribes to no domain event that changed this run (the *changed-domain-event set* = removed-or-renamed events ∪ events with an attribute add/remove/rename):

  ```
  _No internal subscription intersects the changed-domain-event set._
  ```

- **Precise (byte-stable affected)** — when the consumer *does* subscribe (as `internal`) to a domain event that had an attribute add/remove/rename this run, but its Table 3 came out byte-stable (the bound handler does not consume the changed attribute):

  ```
  _Subscribes to changed domain event(s) `<EventName>`, but Table 3 is byte-stable (the bound handler does not consume the changed attribute)._
  ```

  List every such event in `<EventName>` — comma-separated, each backticked. This precise form is the **only** place the byte-stable-affected case is surfaced; there is no corresponding `Warnings:` entry.

A newly-tracked consumer (no HEAD blob) is `unaffected` too and uses whichever of the two forms applies (precise when it subscribes to a changed event, standard otherwise); a separate "first-run consumer" warning marks it as freshly generated rather than updated. Pick exactly one form; never mix them, never add other bullets to an `unaffected` block.

### `needs-init` block body

The consumer has no on-disk spec; the writer renders the *advisory* shape that captures everything the operator needs to initialize the consumer without re-reading the commands diagram. Bullets, in this fixed order:

- **`Spec:`** — the literal italic line `_not yet created_` (no path to render — the spec does not exist).
- **`Commands diagram declaration:`** — `` `%% Messaging - <consumer_name>` in `<dir>/<stem>.commands.md` `` — locates the operator's commands-diagram edit.
- **`Subscriptions declared in commands diagram:`** — a sub-bullet list, one per `Row added` line in the commands-updates report's `### \`<consumer_name>\` (consumer added)` block, copied verbatim (e.g. `` `OrderCommands --() ItemReserved : handles (Inventory, on_item_reserved)` ``). These are *what will be subscribed* once the spec is initialized — not *what is subscribed* (no Table 3 exists yet).
- **`Operator action:`** — exactly one sub-bullet: `Run \`@messaging-spec:code-generator <domain_diagram> <consumer_name>\` to initialize the consumer spec and scaffold its submodule.`

A `needs-init` block carries no pre/post hashes — there is no file to hash.

### `orphaned` block body

The consumer has an on-disk spec but the commands diagram no longer declares it; the writer renders the advisory shape that lets the operator decide preserve-vs-delete. Bullets, in this fixed order:

- **`Spec:`** — the consumer spec path with the literal suffix ` (unchanged this run)` (mirrors the `aborted` block's convention; the spec is unchanged this run by definition).
- **`Commands diagram declaration:`** — the literal italic line `_no \`%% Messaging - <consumer_name>\` block in \`<dir>/<stem>.commands.md\`_`.
- **`Stale subscriptions on the spec:`** — a sub-bullet list, one per Table 2 row currently on the spec, rendered as `` `<EventName>` (<type> · source `<SourceDestination>`) — bound to `<CommandClass>.<command_method>` `` where `<type>` is `internal` or `external` (external rows are listed analogously; the source destination on an external row points to the external publisher). When the spec's Table 2 is empty, render the single sub-bullet `_no subscriptions on the spec_`.
- **`Operator action:`** — exactly two sub-bullets:
  - `Decide whether to preserve the spec file (e.g. hand-authored notes) or delete it; the commands diagram no longer declares this consumer.`
  - `After deciding, run \`@messaging-spec:code-generator <domain_diagram>\` (without a consumer arg) to reconcile the messaging submodule's code side; the orphaned consumer's submodule will be flagged for removal.`

An `orphaned` block carries no pre/post hashes — the spec is unchanged this run; the existing hashes are not delta-relevant.

### Section: Summary

- The eleven lines **Messaging folder**, **Commands diagram**, **Domain updates source**, **Commands-diagram updates source**, **Ops-diagram updates source**, **Consumers discovered**, **Consumers updated**, **Consumers aborted (reconcile commands diagram)**, **Consumers unaffected**, **Consumers needing init**, **Consumers orphaned (every marker dropped)** are always emitted, in that order.
- The identity `updated + aborted + unaffected + needs-init + orphaned = discovered` always holds. `discovered` counts the **union** of (a) on-disk consumer specs, (b) consumer names declared `(consumer added)` by the commands-updates report's `## Messaging Markers` block, and (c) consumer names declared `(consumer added)` by any `## Service:` block's `### Messaging Markers` in the ops-updates report.
- **Domain updates source** value is `_none_` when no domain `updates.md` exists; otherwise the path plus a parenthesised hash (`<dir>/<stem>.domain/updates.md (hash: <sha256>)`).
- **Commands-diagram updates source** value is `_none_` when no `<dir>/<stem>.application/commands-updates.md` exists; otherwise the path plus a parenthesised hash.
- **Ops-diagram updates source** value is `_none_` when no `<dir>/<stem>.application/ops-updates.md` exists; otherwise the path plus a parenthesised hash (`<dir>/<stem>.application/ops-updates.md (hash: <sha256>)`).
- The **Warnings** line is omitted entirely when there are no warnings. When present it introduces a sub-bullet list. Warning categories (each rendered only when applicable; order as listed):
  - Dead subscription: `` `<consumer_name>` subscribes to internal event `<EventName>`, which is no longer emitted by `<AggregateRoot>` (dead subscription — byte-stable spec) `` — emitted when the domain `updates.md` records a `: emits <EventName>` edge removal whose event class survives and some consumer subscribes to it as `internal`. Best-effort: skipped when the domain `updates.md` is missing or carries no parseable relationship-removal entry.
  - Subscription candidate: `` domain event `<EventName>` was added; it is a subscription candidate (declare a `%% Messaging` marker in the commands diagram to consume it) `` — one per `<<Event>>` class added in the domain `updates.md`.
  - First-run consumer: `` `<consumer_name>` is newly tracked (no HEAD blob); its consumer spec was just generated, not updated by this run `` — one per consumer spec with no HEAD blob.
  - Domain updates source missing: `domain updates source not found; all source-delta values fell back to '(unknown source)' and no aborted consumers could be computed` — emitted when `<dir>/<stem>.domain/updates.md` is absent.
  - Commands-diagram updates source missing: `commands-diagram updates source not found; commands-axis source-delta probes skipped and X1/X2 advisory blocks could not be computed` — emitted when `<dir>/<stem>.application/commands-updates.md` is absent.
  - Ops-diagram updates source missing: `ops-diagram updates source not found; ops-axis source-delta probes skipped and ops-declared X1/X2 advisory blocks could not be computed` — emitted when `<dir>/<stem>.application/ops-updates.md` is absent (a present-but-`_None_` ops report does not fire this).
  - Spec-on-disk vs marker-axis race (added): `` consumer `<consumer_name>` reported as added by <report> but spec exists on disk; treating as on-disk-only `` — emitted when a marker-updates report (`commands-updates.md` or `ops-updates.md`) lifecycles a consumer as `(consumer added)` but the spec is already on disk; the writer silently degrades to the on-disk pipeline.
  - Spec-on-disk vs marker-axis race (removed): `` consumer `<consumer_name>` reported as removed by <report> but spec does not exist on disk; treating as absent `` — emitted when a marker-updates report lifecycles a consumer as `(consumer removed)` but no spec is on disk; the writer silently ignores the signal.
- Generated-at timestamps are **not** included — a wall-clock value would break the byte-stability contract.

## `## Operator Actions` section

A consolidated, alphabetical list of human-action items for the run. Always emitted when at least one `needs-init` or `orphaned` consumer exists; **omitted entirely** (no heading, no `_no actions_` placeholder) when both lists are empty.

One bullet per advisory consumer (no status grouping — `needs-init` and `orphaned` interleave alphabetically by consumer name). The bullet shape is exactly one of two literal templates, per the consumer's status:

- `needs-init` consumer `<C>`:

  ```
  - `<C>` — run `@messaging-spec:code-generator <domain_diagram> <C>` to initialize the consumer.
  ```

  Substitute `<domain_diagram>` with the actual diagram path passed to the writer (e.g. `dir/order.md`).

- `orphaned` consumer `<C>`:

  ```
  - `<C>` — preserve or delete `<dir>/<stem>.messaging/<C>.md`, then run `@messaging-spec:code-generator <domain_diagram>` to reconcile the submodule.
  ```

  Substitute `<dir>/<stem>.messaging/<C>.md` with the actual on-disk spec path.

This section is the *consolidated* operator-action list — easy to scan; mirrors the `## Affected Artifacts` footer's "flat dispatch list" ergonomics but for human actions rather than machine ones. Machine consumers (a `/messaging-spec:update-code`) may parse the action templates as a closed vocabulary; the bullet always begins with the backticked consumer name followed by ` — `.

## Source delta phrase grammar

Every `Source delta:` bullet on an `updated` consumer's sub-block carries an **axis tag** identifying which delta report produced the match — or the literal sentinel `(unknown source)` when no probe matches. The grammar:

```
Source delta: [<axis>] <category>: <human_phrase>
```

Where:

- `<axis>` ∈ `{domain, commands-diagram, ops-diagram}`.
- `<category>` is the matched lookup table's category name:
  - `[domain]` axis → `domain-events` (the only domain-axis category that drives an `updated` sub-block).
  - `[commands-diagram]` axis → `messaging-markers` (row add/remove/change inside a `%% Messaging - <C>` block in the commands diagram) or `external-domain-events` (attribute change on an external event the consumer subscribes to).
  - `[ops-diagram]` axis → `messaging-markers` (row add/remove/change inside a `%% Messaging - <C>` block in an ops diagram) or `ops-method` (a bound ops handler method's signature changed). Ops diagrams declare no external `<<Domain Event>>` classes, so there is no `external-domain-events` ops category.
- `<human_phrase>` is a short, deterministic description of the matched delta. Common shapes:
  - `[domain] domain-events: <EventName> attribute <attr> added`
  - `[domain] domain-events: <EventName> attribute <attr> removed`
  - `[domain] domain-events: <EventName> attribute <attr> renamed to <new_attr>` (collapsed form when a remove+add can be paired by the kept sub-block's row delta)
  - `[commands-diagram] messaging-markers: row added (\`<verbatim row text>\`)`
  - `[commands-diagram] messaging-markers: row removed (\`<verbatim row text>\`)`
  - `[commands-diagram] messaging-markers: row changed (\`<verbatim row text>\`)`
  - `[commands-diagram] messaging-markers: row changed (handler binding renamed)` (collapsed form when the `Row changed` describes a `(SourceDestination, on_method)` rename rather than an event-name change)
  - `[commands-diagram] external-domain-events: <EventName> attribute <attr> added/removed/changed`
  - `[ops-diagram] messaging-markers: row added/removed/changed (\`<verbatim row text>\`)`
  - `[ops-diagram] ops-method: \`<method>\` signature changed` (the bound ops handler's parameter list moved, re-deriving Table 3)

Multiple matched phrases for the same sub-block (rare) are joined with `; ` within one axis. When a **marker axis** (commands-diagram or ops-diagram) matches **and** the domain axis would also match, prefer the marker axis — the operator's most-specific edit. The domain axis describes the upstream cause; a marker axis describes the exact diagram-level edit that produced the spec drift. Commands and ops marker probes are disjoint per row in practice (a binding lives in exactly one diagram), so they rarely both fire; if they do, emit both.

When no probe matches the sub-block (defensive fallback only — a malformed delta report, or a sub-block change with no upstream signal), render the literal:

```
Source delta: (unknown source)
```

No axis tag, no category, no phrase. Same literal regardless of which axis was probed.

### Worked example — mixed-axis attribution within one consumer

A single `updated` consumer can carry sub-blocks driven by *different* axes (domain attribute change on one event ⊕ commands-marker row add on another). Per-sub-block tagging is the only way to attribute correctly:

```markdown
### `profile-reconciliation` — updated

- Spec: `dir/order.messaging/profile-reconciliation.md`
- Pre-update hash: a1b2c3...
- Post-update hash: d4e5f6...
- Table 2 refreshed: added: `ProfileUpdated`
- Table 3 sub-blocks regenerated:
  - `**Event:** ProfileSubmitted` (internal · source `Profiles`)
    - Source delta: [domain] domain-events: ProfileSubmitted attribute region added
    - Event Field mappings changed:
      - row `region` ↦ `region` added
    - Low-confidence flags: _none_
  - `**Event:** ProfileUpdated` (internal · source `Profiles`)
    - Source delta: [commands-diagram] messaging-markers: row added (`OrderCommands --() ProfileUpdated : handles (Profiles, on_profile_updated)`)
    - Event Field mappings changed:
      - row `profile_id` ↦ `profile_id` added
      - row `updated_at` ↦ `updated_at` added
    - Low-confidence flags: _none_
```

Notice that the two sub-blocks carry **different axis tags** — one `[domain]`, one `[commands-diagram]` — but live inside one `updated` consumer block. The `Table 2 refreshed:` line above them is itself a commands-diagram-driven artifact (the operator added a row to `%% Messaging - profile-reconciliation`); it precedes the sub-block list and is rendered verbatim from the Table 2 row diff.

## `## Affected Artifacts` computation

A flat dispatch table. The code updater walks it top-to-bottom. Only `updated` consumers contribute rows; `aborted`, `unaffected`, `needs-init`, and `orphaned` consumers contribute nothing.

`needs-init` and `orphaned` consumers contribute **zero rows** — they surface in `## Operator Actions` only. The footer's action vocabulary remains `{add, modify, remove}`; `init` and `delete-or-preserve` are operator-decisions, not code-updater dispatch entries. (A `needs-init` consumer has no on-disk artifact yet; an `orphaned` consumer's existing artifacts are byte-stable on the messaging-update axis and their cleanup is operator-decision-gated.)

For each `updated` consumer `C` (consumer name `<C>`, snake form `<C_snake>` = `<C>` with every `-` replaced by `_`), emit, in this order:

1. `messaging/<C_snake>/handlers.py | modify | <C>`
2. `tests/integration/messaging/<C_snake>/test_<C_snake>_handlers.py | modify | <C>`

On the messaging-update axes only `modify` ever appears — an internal-event attribute change or a Table 2 row add re-renders an existing handler's call kwargs and that handler's test; it never adds or removes a handler (handler add/remove is a consumer-lifecycle concern, surfaced as `needs-init` or `orphaned` in `## Operator Actions`), and `dispatcher.py`, `events.py`, `constants.py`, and the `containers.py` / `entrypoint.py` / `__main__.py` wiring are byte-stable on these axes for an existing consumer. The per-handler granularity (which handler inside `handlers.py`) lives in the consumer's `## Consumer Changes` body (the regenerated sub-block list), not in the footer.

### Row ordering

Consumers in the table follow the same alphabetical order as `## Consumer Changes`. Within a consumer's two rows, `handlers.py` precedes the test module.

### Action vocabulary

The `Action` column is a closed set: `add`, `modify`, `remove`. On the messaging-update axes only `modify` is emitted. The `Driving consumer` column is the consumer name in backticks — the same anchor used as the H3 heading in `## Consumer Changes`.

The table header (`| Path | Action | Driving consumer |` plus the divider row) is always emitted. When no consumer is `updated`, the table has no data rows.

## Hash format

All hashes in this report are SHA256 of UTF-8 file content, rendered in **lowercase hex**, full **64-character** form.

When a hash cannot be computed (file missing or unreadable), render the value literally as `(none)`. Never substitute zeros.

## Determinism contract

- Byte-stable inputs (working-tree consumer specs, HEAD consumer specs, sibling `<dir>/<stem>.domain/updates.md`, sibling `<dir>/<stem>.application/commands-updates.md`, sibling `<dir>/<stem>.application/ops-updates.md`, sibling `<dir>/<stem>.commands.md`) → byte-stable report.
- Re-running the writer with no new changes produces a report whose every consumer is `unaffected` (or the appropriate advisory if a commands or ops diagram declared/dropped a marker), with an empty Affected Artifacts row list and the same sentinel hash triple.
- Section ordering, consumer ordering (alphabetical, no status grouping), sub-block ordering (Table 3 order — `external` alphabetical then `internal` alphabetical, so in practice alphabetical by Event Name), and within-block bullet order are absolute. No source-defined deviation.
- The `## Operator Actions` section is omitted entirely when its bullet list would be empty — never replaced with a placeholder.
- The Summary deliberately excludes a `Generated at` line.
